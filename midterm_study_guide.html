A 문제 — Heap (힙)

총 21점

힙(Heap) 핵심 개념 — 코드 짜기 전에 반드시 이해

📐 Min-Heap 두 가지 성질

  • 힙 순서(Heap-Order): 루트를 제외한 모든 노드 v에 대해 key(v) ≥ key(parent(v)). 즉, 부모가 항상 자식보다 작거나 같아야 한다. 결과적으로 루트 = 최솟값.
  • 완전 이진 트리(Complete Binary Tree): 마지막 레벨을 제외한 모든 레벨이 꽉 차 있고, 마지막 레벨은 반드시 왼쪽부터 채워진다. 이 덕분에 배열로 표현 가능.
  • 배열 인덱스 공식: 인덱스 j → 부모: (j-1)//2, 왼쪽 자식: 2j+1, 오른쪽 자식: 2j+2
  • upheap vs downheap: 원소를 삽입(add)하면 끝에 추가 후 upheap으로 올려 복구. 최솟값을 삭제(remove_min)하면 루트↔마지막 교환 후 downheap으로 내려 복구.

a1 · _upheap(self, j)

인덱스 j의 원소를 위로 올려 힙 순서 복구 · 재귀
중간 · 8점 일부

📐 동작 원리 — 단계별

  • 종료 조건 ①: j == 0이면 이미 루트. 더 올라갈 곳이 없으므로 아무것도 하지 않고 종료. (if 조건에 j > 0이 포함되어 있어 자동 처리)
  • 종료 조건 ②: data[j] ≥ data[parent]이면 이미 힙 순서가 만족되므로 종료. 더 올릴 필요 없음.
  • swap: 두 조건이 모두 위반(j > 0 이고 나 < 부모)이면, j번째와 parent번째 원소를 교환.
  • 재귀 호출: swap한 후, 이제 원소가 parent 위치로 이동했으므로 _upheap(parent)를 호출해 계속 위로 올라감.

💻 코드

🙈 코드를 가렸습니다버튼을 눌러 다시 보세요
def _upheap(self, j):
    parent = self._parent(j)                        # 부모 인덱스 계산
    if j > 0 and self._data[j] < self._data[parent]:
        self._swap(j, parent)                       # j ↔ 부모 교환
        self._upheap(parent)                     # 부모 위치에서 재귀 ← 핵심!

🧠 암기 포인트

  • 조건 두 개: j > 0 (루트가 아님) AND 나 < 부모 (힙 위반)
  • swap 후 재귀 인자는 반드시 parent. j를 넣으면 무한루프!
  • add()를 호출하면 → append 후 → upheap(len-1) 순서로 이어짐

⚠️ 자주 틀리는 부분

  • 재귀 호출: self._upheap(j) ← 틀림. 반드시 self._upheap(parent)
  • 비교 방향: data[j] > data[parent] ← 반대로 틀림. <이어야 올라감

a1 · _downheap(self, j)

인덱스 j의 원소를 아래로 내려 힙 순서 복구 · 재귀
중간 · 8점 일부

📐 동작 원리 — 단계별

  • 종료 조건: 왼쪽 자식이 없으면(_has_left(j)가 False) 리프 노드 → 더 내려갈 곳 없음. 종료. 참고: 완전 이진 트리에서 오른쪽 자식은 왼쪽 자식이 있어야만 존재 가능.
  • 작은 자식 선택 ①: 일단 small_child = left로 가정. 왼쪽이 후보.
  • 작은 자식 선택 ②: 오른쪽 자식도 있고(_has_right) data[right] < data[left]이면 오른쪽이 더 작은 것 → small_child = right로 교체.
  • swap 조건: data[small_child] < data[j]이면 자식이 부모보다 작으므로 힙 위반 → swap.
  • 재귀 호출: swap 후 원소가 small_child 위치로 내려갔으므로 _downheap(small_child) 호출해 계속 내려감.

💻 코드

🙈 코드를 가렸습니다버튼을 눌러 다시 보세요
def _downheap(self, j):
    if self._has_left(j):                           # 왼쪽 자식이 없으면 리프 → 종료
        left = self._left(j)
        small_child = left                          # ① 왼쪽을 일단 최소 후보로 가정

        if self._has_right(j):
            right = self._right(j)
            if self._data[right] < self._data[left]:
                small_child = right                 # ② 오른쪽이 더 작으면 교체

        if self._data[small_child] < self._data[j]:  # ③ 자식이 부모보다 작으면
            self._swap(j, small_child)
            self._downheap(small_child)             # ④ 내려간 위치에서 재귀 ← 핵심!

🧠 암기 포인트

  • 흐름: 왼 가정 → 오 비교 → 작은쪽 swap → small_child로 재귀
  • 왼쪽 자식이 없으면 if 전체를 안 타므로 자동 종료
  • 재귀 인자는 반드시 small_child. j를 넣으면 무한루프!

⚠️ 원본 a1.py 버그 주의!

  • self._data.left(j) ← 틀린 코드. 올바른 건 self._left(j)
  • 비교 방향 self._data[j] < self._data[min] ← 반대로 틀림. self._data[small_child] < self._data[j]가 맞음

❓ 이해 확인

  1. downheap에서 왜 오른/왼쪽 자식끼리 먼저 비교하고, 그 결과를 부모와 비교하는가?
  2. 오른쪽 자식은 있고 왼쪽 자식은 없는 경우가 완전 이진 트리에서 가능한가?
정답 확인
1. Min-Heap에서 부모보다 작은 자식 중 더 작은 쪽과 swap해야 힙 순서가 계속 유지됨. 만약 더 큰 자식과 swap하면, 그 더 큰 자식이 더 작은 형제보다 커져 버려 또다시 힙 위반이 생김.

2. 불가능. 완전 이진 트리에서 오른쪽 자식은 항상 왼쪽 자식이 먼저 채워진 뒤에 생긴다. 따라서 has_left를 먼저 확인하는 구조가 항상 옳다.

a2 · add(self, value)

힙에 원소 삽입 후 upheap으로 힙 순서 복구
중간 · 8점 일부

📐 동작 원리

append(value)→ 끝 인덱스 = len-1→ _upheap(len-1)→ 힙 복구 완료

📐 왜 이렇게 하는가?

  • append: 완전 이진 트리의 다음 빈 자리 = 배열의 맨 끝. append하면 트리 구조 유지.
  • 끝 인덱스: append 후에 len(self._data) - 1이 새 원소의 인덱스.
  • upheap: 끝에 추가된 원소가 힙 순서를 위반할 수 있으므로 위로 올려 복구.

💻 코드

🙈 코드를 가렸습니다버튼을 눌러 다시 보세요
def add(self, value):
    self._data.append(value)               # 배열 맨 끝에 추가
    self._upheap(len(self._data) - 1)      # 새 원소 인덱스에서 upheap

🧠 암기 포인트

  • append → upheap(len-1), 딱 두 줄이 전부
  • 인덱스 계산: append 후에 len-1 계산해야 함 (순서 바꾸면 틀림)

a2 · remove_min(self) → int

최솟값(루트) 제거 후 downheap으로 힙 순서 복구
중간 · 8점 일부

📐 동작 원리 — 왜 이렇게 복잡하게 하는가?

  • 예외 처리: 비어있으면 Exception. 이후 로직이 비어있는 상태를 가정하지 않음.
  • swap(0, 끝): 루트를 바로 제거하면 완전 이진 트리 구조가 깨짐. 대신 가장 마지막 원소(제거해도 구조 유지)와 먼저 교환.
  • pop(): 이제 배열 맨 끝에 있는 것이 원래 루트(최솟값). 안전하게 제거.
  • is_empty 체크: 원소가 딱 1개였다면 pop() 후 배열이 완전히 빔. 이 경우 downheap(0)을 호출하면 인덱스 오류 발생. 반드시 체크!
  • downheap(0): 루트 자리에 올라온 (원래 마지막이던) 원소가 힙 순서를 위반할 수 있으므로 아래로 내려 복구.

💻 코드

🙈 코드를 가렸습니다버튼을 눌러 다시 보세요
def remove_min(self):
    if self.is_empty():
        raise Exception("Heap is empty")           # 예외 처리

    self._swap(0, len(self._data) - 1)          # 루트 ↔ 마지막 교환
    item = self._data.pop()                    # 마지막(=원래 루트=최솟값) 제거

    if not self.is_empty():                   # ← 이 체크가 없으면 1개 남았을때 터짐!
        self._downheap(0)                       # 새 루트를 아래로 내리기
    return item

🧠 암기 포인트

  • 5단계 순서: 예외처리 → swap(0,끝) → pop → is_empty체크 → downheap(0) → return
  • not self.is_empty() 체크를 빠뜨리면 원소 1개짜리 힙에서 오류!
  • 반환값 = pop()한 결과 = 원래 루트 = 최솟값

❓ 이해 확인

  1. 루트를 바로 제거하지 않고 마지막 원소와 swap하는 이유는?
  2. downheap 전에 is_empty를 체크하는 이유는?
  3. 힙 [1, 3, 2, 5, 4]에서 remove_min() 후 힙 상태는 어떻게 되는가?
정답 확인
1. 힙은 완전 이진 트리 구조를 유지해야 함. 루트를 바로 제거하면 트리 중간에 구멍이 생김. 마지막 원소와 swap하면 완전 이진 트리에서 마지막 리프만 제거하는 것이므로 구조가 유지됨.

2. 원소가 정확히 1개일 때 remove_min을 호출하면 pop() 후 배열이 완전히 비어버림. 빈 배열로 downheap(0)을 호출하면 self._data[0] 접근 자체가 IndexError.

3. swap(0, 4) → [4,3,2,5,1] → pop() → [4,3,2,5] → downheap(0): 4와 자식들 중 최솟값(2)과 swap → [2,3,4,5]. 결과: [2,3,4,5]

a3 · heap_sort(arr) → list

Heap 클래스를 이용한 오름차순 정렬
쉬움 · 5점

📐 동작 원리

  • Heap 생성: 빈 Heap 객체를 만든다.
  • 전부 삽입: arr의 모든 원소를 heap.add()로 삽입. add()가 힙 순서를 자동 유지하므로 루트 = 항상 최솟값.
  • 전부 꺼내기: heap.remove_min()을 반복 호출. Min-Heap이므로 항상 현재 최솟값이 나옴 → 호출 순서대로 오름차순 정렬.
  • 반환: 새로 만든 sorted_arr를 반환. 원본 arr은 변경하지 않음.

💻 코드

🙈 코드를 가렸습니다버튼을 눌러 다시 보세요
def heap_sort(arr):
    heap = Heap()
    for i in arr:
        heap.add(i)                             # 전부 삽입
    sorted_arr = []
    while not heap.is_empty():
        sorted_arr.append(heap.remove_min())  # 전부 꺼내기 → 오름차순
    return sorted_arr

🧠 암기 포인트

  • "전부 add → 전부 remove_min", 이게 heap sort의 전부
  • 원본 arr은 건드리지 않음. 새 리스트를 만들어 반환
  • 시간복잡도: O(n log n) — add 하나에 O(log n), 총 n번

B 문제 — Binary Tree (이진 트리)

총 34점

공통 사항 — 예제 트리 & find_node 패턴

📐 예제 트리 (b문제 전체에서 사용)

        1            ← 깊이 0 (루트)
      /   \
     9     8         ← 깊이 1
    / \   /
   6   2 3           ← 깊이 2 (리프들)

📐 find_node 헬퍼 — b3, b4에서 반복 사용

key를 가진 노드를 재귀로 탐색. 없으면 None 반환. b3, b4 코드에서 내부 함수로 그대로 쓰인다.

🙈 코드를 가렸습니다버튼을 눌러 다시 보세요
def find_node(node, key):
    if node is None: return None          # base case: 없음
    if node.key == key: return node       # 찾음!

    left_result = find_node(node.left, key)
    if left_result is not None:
        return left_result                  # 왼쪽에서 찾았으면 바로 반환

    return find_node(node.right, key)     # 아니면 오른쪽 탐색
  • 왼쪽에서 찾았으면 not None 체크 후 즉시 반환 → 오른쪽 탐색 생략 (효율)
  • 맨 마지막 줄 return find_node(node.right, key)에서 return 빠뜨리면 None 반환됨!

b1 · preorder(tree) → list

전위 순회: 부모 → 왼쪽 자식 → 오른쪽 자식
쉬움 · 5점

📐 동작 원리

  • 현재 노드 처리: 현재 노드의 key를 먼저 result에 추가 (Pre = 먼저).
  • 왼쪽 재귀: 왼쪽 서브트리를 동일한 방식으로 전위 순회.
  • 오른쪽 재귀: 오른쪽 서브트리를 동일한 방식으로 전위 순회.

예제 결과: [1, 9, 6, 2, 8, 3] — 루트(1)부터 시작, 왼쪽 서브트리 전체(9,6,2) 후 오른쪽(8,3)

💻 코드

🙈 코드를 가렸습니다버튼을 눌러 다시 보세요
def preorder(tree):
    result = []

    def _recur(node):
        if node is not None:
            result.append(node.key)   # ① 나 먼저 (Pre = 먼저)
            _recur(node.left)          # ② 왼쪽
            _recur(node.right)         # ③ 오른쪽

    _recur(tree.root())
    return result

🧠 암기 포인트

  • Pre(전위) = 먼저(前): append가 두 재귀 호출 앞에 있다
  • 순서: 나 → 왼 → 오
  • node가 None이면 if 안을 실행하지 않으므로 자동 base case 처리

b2 · postorder(tree) → list

후위 순회: 왼쪽 자식 → 오른쪽 자식 → 부모
쉬움 · 5점

📐 동작 원리

  • 왼쪽 재귀: 왼쪽 서브트리를 완전히 다 처리.
  • 오른쪽 재귀: 오른쪽 서브트리를 완전히 다 처리.
  • 현재 노드 처리: 자식들이 모두 끝난 뒤에야 현재 노드의 key를 추가 (Post = 나중).

예제 결과: [6, 2, 9, 3, 8, 1] — 리프(6,2)부터, 루트(1)가 맨 마지막

💻 코드

🙈 코드를 가렸습니다버튼을 눌러 다시 보세요
def postorder(tree):
    result = []

    def _recur(node):
        if node is not None:
            _recur(node.left)          # ① 왼쪽
            _recur(node.right)         # ② 오른쪽
            result.append(node.key)   # ③ 나 나중에 (Post = 나중)

    _recur(tree.root())
    return result

🧠 암기 포인트

  • Post(후위) = 나중(後): append가 두 재귀 호출 뒤에 있다
  • preorder와 비교: append의 위치만 다름. 나머지 코드는 완전히 동일!

❓ 이해 확인

  1. preorder vs postorder — 코드에서 다른 부분은 딱 하나, 뭔가?
  2. inorder(중위 순회, 왼→나→오)는 코드를 어떻게 바꾸면 되는가?
정답 확인
1. result.append(node.key)의 위치. preorder는 재귀 전, postorder는 재귀 후.

2. 왼쪽 재귀 후, 오른쪽 재귀 전에 append를 넣으면 됨:
_recur(node.left) → append(node.key) → _recur(node.right)

b3 · depth(tree, key) → int

노드를 찾고, parent 포인터를 따라 루트까지 올라가며 깊이 카운트
중간 · 8점

📐 동작 원리 — 2단계 구조

  • [1단계] find_node: 재귀로 key에 해당하는 노드 탐색. 없으면 즉시 return -1.
  • [2단계] parent 타고 올라가기: 찾은 노드에서 .parent를 계속 따라 올라감. 루트의 parent는 None이므로 while 조건: curr.parent is not None이면 계속 올라감.
  • 카운트: 올라갈 때마다 d += 1. 루트에 도달하면 d = 해당 노드의 깊이.

예시: 노드 3의 깊이 = 2 (3→8→1, parent 2번 타고 올라감)

def depth(tree, key):
    def find_node(node, key):           # 1단계: 노드 탐색
        if node is None: return None
        if node.key == key: return node
        left_result = find_node(node.left, key)
        if left_result is not None: return left_result
        return find_node(node.right, key)

    target = find_node(tree.root(), key)
    if target is None: return -1         # 못 찾으면 -1

    d = 0                               # 2단계: parent 타고 올라가기
    curr = target
    while curr.parent is not None:
        curr = curr.parent
        d += 1
    return d

🧠 암기 포인트

  • "찾고 → parent 타고 카운트" 두 단계
  • while 조건: curr.parent is not None. 루트(parent=None)에 도달하면 종료.
  • 이동 순서: curr = curr.parent 먼저, d += 1 나중 (순서 바뀌면 안 됨)

⚠️ 자주 틀리는 부분

  • target is None 체크 후 return -1을 find_node 바로 다음에 써야 함. 안 쓰면 None.parent 접근 → AttributeError!
  • 루트 노드 자신의 깊이 = 0. while이 한 번도 실행되지 않아 d=0 그대로 반환 — 정확한 동작.

b4 · uncle(tree, key) → int

삼촌 = 부모의 형제 노드 key 반환. 없으면 -1
중간 · 8점

📐 동작 원리 — 3단계 구조

  • [1단계] 노드 찾기: find_node로 key에 해당하는 target 탐색.
  • [2단계] p(부모)와 g(조부모) 확인: p = target.parent, g = p.parent. p가 None이면 target이 루트(삼촌 불가능), g가 None이면 p가 루트(삼촌 불가능) → 둘 다 return -1.
  • [3단계] 삼촌 찾기: p가 g의 왼쪽 자식(g.left == p)이면 삼촌 = g.right, 아니면 삼촌 = g.left. 해당 노드가 None이면 return -1.
   g(1)               ← 조부모
  /    \
p(9)  uncle(8)        ← p가 왼쪽이면 uncle = g.right

def uncle(tree, key):
    def find_node(node, key):
        if node is None: return None
        if node.key == key: return node
        left_result = find_node(node.left, key)
        if left_result is not None: return left_result
        return find_node(node.right, key)

    target = find_node(tree.root(), key)
    p = target.parent
    if p is None: return -1             # target이 루트 → 부모 없음

    g = p.parent
    if g is None: return -1             # p가 루트 → 조부모 없음

    if g.left == p:                    # p가 g의 왼쪽 자식이면
        return g.right.key if g.right is not None else -1
    else:                              # p가 g의 오른쪽 자식이면
        return g.left.key if g.left is not None else -1

🧠 암기 포인트

  • 삼촌 = 조부모(g)의 반대쪽 자식
  • -1 반환 케이스 3가지: ① p is None (target=루트) ② g is None (p=루트) ③ 반대쪽 자식이 None
  • g.left == p로 p가 왼쪽인지 확인 → 삼촌은 g.right

  2. 예제 트리에서 노드 3의 uncle은?
  3. -1을 반환해야 하는 3가지 케이스를 정확히 말하라.
정답 확인
1. -1. 노드 9의 부모 = 1(루트), 1의 부모(g) = None → 조부모가 없으므로 삼촌 불가능.

2. 9. 3의 부모 = 8, 8의 부모(g) = 1. g(1)에서 8은 오른쪽 자식이므로 삼촌 = g.left = 9.

3. ① target.parent is None (target이 루트)   ② p.parent is None (p가 루트)   ③ g의 반대쪽 자식이 None (형제가 없음)

b5 · max_path_sum(tree) → int

루트 → 리프까지의 경로 중 key 합이 최대인 값 반환
중간 · 8점

📐 동작 원리

  • Base case: node가 None이면 return 0. 존재하지 않는 경로의 기여는 0.
  • 왼/오른쪽 재귀: 각 서브트리의 "루트→리프 최대 합"을 재귀로 계산.
  • 현재 노드 기여: node.key + max(left_sum, right_sum). 더 큰 합의 경로를 선택하고 현재 key를 더함.
  • 빈 트리 처리: 함수 맨 위에서 tree.root is None이면 0 반환.
recur(6)=6,  recur(2)=2
recur(9) = 9 + max(6, 2) = 15
recur(3)=3,  recur(None)=0
recur(8) = 8 + max(3, 0) = 11
recur(1) = 1 + max(15, 11) = 16  ✓

def max_path_sum(tree):
    def _recur(node):
        if node is None: return 0          # base case: 없으면 0 (−1 아님!)

        left_sum  = _recur(node.left)
        right_sum = _recur(node.right)

        return node.key + max(left_sum, right_sum)  # 더 큰 쪽 경로 선택

    if tree.root is None: return 0        # 빈 트리이면 0
    return _recur(tree.root())

🧠 암기 포인트

  • 핵심 공식: 내 key + max(왼쪽합, 오른쪽합)
  • base case는 반드시 return 0. -1이면 max()가 잘못된 값을 선택함!
  • 빈 트리 체크: tree.root is None (속성 접근, 메서드 호출 아님)

⚠️ 자주 틀리는 부분

  • base case return -1 ← 절대 안 됨! 리프의 자식들이 모두 -1이면 max(-1,-1)=-1이 더해져서 오답.
  • 빈 트리 체크: tree.root() ← 메서드 호출이면 오류 가능. tree.root 속성으로 확인.

❓ 이해 확인

  1. base case를 return -1로 쓰면 왜 틀린 결과가 나오는가? 구체적 예시로 설명하라.
  2. 루트 하나만 있는 트리(key=7)의 max_path_sum은?
정답 확인
1. 리프 노드(예: key=6)의 경우 left, right가 모두 None → recur(None) = -1. 그러면 6 + max(-1, -1) = 6 + (-1) = 5가 됨. 실제 경로 합(6)보다 작아져서 오답. return 0이어야 6 + max(0, 0) = 6으로 정확함.

2. 7. _recur(root) = 7 + max(recur(None), recur(None)) = 7 + max(0, 0) = 7

🌿 힙 함수 패턴

  • _upheap: j>0 AND 나<부모 → swap → upheap(parent)
  • _downheap: 왼 가정 → 오 비교 → swap → downheap(small_child)
  • add: append → upheap(len-1)
  • remove_min: swap(0,끝) → pop → is_empty체크 → downheap(0) → return
  • heap_sort: 전부 add → 전부 remove_min

🌳 트리 함수 패턴

  • preorder: append → 왼재귀 → 오재귀 (나 먼저)
  • postorder: 왼재귀 → 오재귀 → append (나 나중)
  • depth: find_node → parent 타고 올라가며 카운트
  • uncle: p=target.parent, g=p.parent → g의 반대쪽. -1 케이스 3가지!
  • max_path_sum: 나 + max(왼,오). None이면 0

⚠️ 절대 틀리면 안 되는 것

  • upheap 재귀: parent (j 아님)
  • downheap 재귀: small_child (j 아님)
  • remove_min: downheap 전 is_empty 체크
  • max_path_sum base: return 0 (-1 아님)
  • uncle -1 케이스: 3가지 모두
  • find_node 마지막 줄: 반드시 return 붙이기

🔁 공통 find_node 패턴

  • node is None → return None
  • node.key == key → return node
  • 왼쪽 재귀 결과 not None → 즉시 return
  • 마지막: return find_node(right)
